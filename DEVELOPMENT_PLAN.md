## Этап 1. API-слой и загрузка данных

### 1.1 Единый API-клиент

- [ ] Создать `renderer/js/api.js` — обёртка над fetch с:
  - базовым URL из main process (IPC)
  - заголовком `Authorization: Bearer <token>`
  - автоматическим refresh при 401
  - обработкой сетевых ошибок
- [ ] IPC: `get-access-token`, `get-server-url`, `api-request` (опционально — проксировать запросы через main)

### 1.2 Реализация видов по API

| Вид | API | Приоритет |
|-----|-----|-----------|
| Мероприятия — Организация | `POST /api/events/org/list`, `count` | 1 |
| Мероприятия — Участие | `POST /api/events/part/list`, `count` | 1 |
| Сотрудники | `GET /api/employees`, `all`, `search` | 2 |
| Ученики | `GET /api/students`, `search` | 2 |
| Расписание | `GET /api/schedule`, `by-teacher`, `by-group` | 2 |
| Бронь | `GET /api/rent/*` | 3 |
| Уведомления | `PUT /api/events/org/notifications`, `part/notifications` | 3 |
| Документы | по документации | 3 |
| Выгрузка | по документации | 3 |
| Профиль | `GET /api/employees/:id` или данные из user | 2 |

### 1.3 Состояния загрузки

- [ ] Loading (скелетон или спиннер)
- [ ] Empty (пустой список)
- [ ] Error (повторная попытка)
- [ ] Success (данные)

## Этап 2. Валидация и верификация

### 2.1 Валидация форм

| Форма | Поля | Правила |
|-------|------|---------|
| auth-url | serverUrl | URL, обязательное, протокол http/https |
| auth-login | login, password | Обязательные, login — не пустой после trim |
| Фильтры списков | даты, поиск | Формат дат, длина строк |

- [ ] Утилита `renderer/js/validation.js`:
  - `isValidUrl(url)` — URL с http/https
  - `required(value)` — не пусто
  - `maxLength(str, n)`
  - `dateFormat(str)` — YYYY-MM-DD
- [ ] Показ ошибок под полями, блокировка submit при невалидных данных

### 2.2 Верификация ответов API

- [ ] Проверка `success` в ответе
- [ ] Проверка структуры `data` (типы, обязательные поля)
- [ ] Обработка кодов ошибок (см. API_DOCUMENTATION.md §13)
- [ ] Опционально: схема валидации (Zod, Joi) для критичных ответов

### 2.3 Безопасность и сессия

- [ ] Refresh токена при 401 перед повторным запросом
- [ ] Выход при истечении refresh-токена (редирект на auth-login)
- [ ] Не логировать пароли и токены
- [ ] Санитизация ввода (XSS) при отображении пользовательских данных

### 2.4 Валидация main process

- [ ] Проверка URL при `server-url-set`
- [ ] Проверка структуры `auth-success` (наличие accessToken, user)
- [ ] Обработка ошибок `apiRequest` (таймаут, ECONNREFUSED, EACCES)

## Этап 3. Тестирование

### 3.1 Unit-тесты (Jest)

**Зависимости:** `jest`, `jest-environment-node`

- [ ] `validation.js` — тесты для `isValidUrl`, `required`, `dateFormat`
- [ ] `api.js` — моки fetch, проверка формирования заголовков и refresh
- [ ] Утилиты (если появятся) — форматирование дат, пагинация

**Структура:**
```
__tests__/
  validation.test.js
  api.test.js
```

### 3.2 Интеграционные тесты main process

- [ ] `main.js` — мок Store, проверка `tryAutoLogin` (успех/неуспех)
- [ ] `apiRequest` — мок https.request, проверка вызовов при разных URL

### 3.3 E2E-тесты (Playwright + Electron)

**Зависимости:** `@playwright/test`, `playwright`

- [ ] Сценарий: ввод URL → ввод логина/пароля → главное окно
- [ ] Сценарий: автовход (мок store с credentials) → главное окно без формы
- [ ] Сценарий: переключение видов меню
- [ ] Сценарий: выход (если будет реализован)

**Конфиг:** `playwright.config.js` с `use: { launchOptions: { executablePath: 'node_modules/.bin/electron' } }` или аналог.

### 3.4 Ручное тестирование (чек-лист)

- [ ] Первый запуск: auth-url → auth-login → main
- [ ] Повторный запуск с «Запомнить»: автовход
- [ ] Повторный запуск без «Запомнить»: форма входа
- [ ] localhost и внешний сервер
- [ ] Самоподписанный сертификат
- [ ] Сеть отключена: понятное сообщение об ошибке
- [ ] Неверный пароль: сообщение на форме
- [ ] Сессия истекла: сообщение «Сессия истекла»
